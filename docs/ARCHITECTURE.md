# Architecture

This document captures the high-level design of the **Tansiq Information Portal**
(`criminal-archive`). It is intentionally short — the codebase is the source of
truth, this document explains *why*.

---

## 1. Goals

1. Build a **public, community-driven archive** of crimes (especially sensitive
   cases such as sexual violence) with verified evidence, perpetrator and
   victim profiles, and case timelines.
2. Behave like the **Internet Archive**: anyone can browse and download
   *published* material without an account.
3. Submissions go through **two gates** — a verification team (evidence) and
   admin review (publication) — before becoming public.
4. Operate on **modest infrastructure** while serving heavy read traffic and
   storing large evidence files.

## 2. Non-goals (for now)

- Real-time messaging or chat between users.
- Mobile native apps. (PWA is fine; native is out of scope until v2.)
- Anonymous submissions without an account. Submitters must register and be
  approved; their identity stays admin-visible only.

---

## 3. High-level shape

```
                 ┌─────────────────────────────────────────┐
                 │            Cloudflare (CDN/WAF)         │
                 └────────────┬─────────────┬──────────────┘
                              │             │
          static SPA (HTML/JS/CSS)     /api/v1/*   public file URLs
                              │             │             │
                  ┌───────────▼──┐   ┌──────▼──────┐   ┌──▼─────────────┐
                  │   nginx      │   │  Go API     │   │ Cloudflare R2  │
                  │ (frontend)   │   │ (chi+pgx)   │   │ (S3 compatible)│
                  └──────────────┘   └──────┬──────┘   └────────────────┘
                                            │
                                     ┌──────▼──────┐
                                     │ PostgreSQL  │
                                     │     18      │
                                     └─────────────┘
```

- **Frontend** is a static React/Vite SPA, served from any CDN/edge cache.
- **Backend** is a single Go binary exposing a JSON HTTP API and signing R2
  upload/download URLs. No long-running queues yet (jobs are inline).
- **Storage** for evidence files is Cloudflare R2 (S3-compatible). Local dev
  uses MinIO with the same API.
- **Database** is PostgreSQL 18.

## 4. Backend layout (`backend/`)

```
cmd/api/                main()  — wires config, db, router
internal/
  config/               env loader (.env + os.LookupEnv)
  db/                   pgxpool wrapper
  auth/      [planned]  argon2/bcrypt + JWT + RBAC middleware
  storage/   [planned]  R2/S3 client + presigned URLs + rename helpers
  handlers/  [planned]  HTTP handlers grouped by resource
  middleware/[planned]  logging, rate-limit, request-id, RBAC
  models/    [planned]  domain structs (no DB-tagged models — pgx scans)
  repository/[planned]  SQL queries (sqlc-friendly)
  service/   [planned]  business logic (case publish, file rename, etc.)
  router/               route registration
migrations/             *.up.sql / *.down.sql
seed/                   reference data (countries, BD locations, crime types)
```

### Key conventions

- One **domain → one handler file → one repository file → one service file**.
- Handlers do *only* I/O: parse → call service → encode response.
- Services own transactions and orchestration.
- Repositories own SQL. No ORM; we use `pgx` directly.
- All timestamps in UTC; DB columns are `TIMESTAMPTZ`.

## 5. Data model highlights

See [`backend/migrations/0001_init.up.sql`](../backend/migrations/0001_init.up.sql)
for the authoritative schema. Notable points:

- **`persons`** is shared between victims, accused and witnesses. A person can
  appear in many cases through `case_persons (case_id, person_id, role)`. This
  is what makes "click on a person → see all their cases" a single SQL query.
- **`cases`** carries a human-friendly `case_number` like `TIP-2026-00045`,
  generated by an atomic `INCREMENT` on `case_number_counters(year)`.
- **`case_attachments.kind`** is one of `public | hidden | internal`. The
  public API filters out `hidden` and `internal` for non-admin users.
- **Locations** are normalised: `country → division → district → upazila`. For
  Bangladesh we ship the full hierarchy; for other countries we expose only
  `country` + a free-text `location_text`.
- **`audit_logs`** captures every privileged action (approvals, edits to
  someone else's submission, deletions). Append-only, never updated.
- **`sessions`** stores *hashed* refresh tokens; access tokens are short-lived
  JWTs not stored server-side.
- `pg_trgm` indexes are added on names and titles for fast fuzzy search.

### Submission state machine

```
draft → pending_review → in_verification → approved → published
                    │            │              │
                    └────────────┴──────────────┴──→ rejected | archived
```

A case becomes `published` only after **both** verification and admin
approval. Public reads filter by `status = 'published'`.

## 6. File storage & naming

- Evidence is uploaded **directly to R2** via a presigned PUT URL issued by
  the API. This keeps large uploads off the Go server.
- Once R2 confirms the upload, the API records a `case_attachments` row and
  computes the canonical `stored_filename`:

  ```
  TIP-{YYYY}-{SEQ:05d}_{kind}_{idx:02d}.{ext}
  ```

  Examples:
  - `TIP-2026-00045_evidence_03.jpg`
  - `TIP-2026-00045_hidden_01.pdf`
  - `TIP-2026-00045_internal_02.docx`

  The original filename is preserved in `original_filename` for reference.
- The R2 object key is rewritten to match `stored_filename` either by a
  server-side `CopyObject` + `DeleteObject`, or — preferred — by issuing the
  presigned URL with the canonical key in the first place (we resolve the
  next sequence number before issuing the URL).
- `public` attachments are served directly from R2 through a Cloudflare
  custom domain with long cache headers. `hidden` and `internal` attachments
  are served only via short-lived presigned download URLs that the API
  generates per request after RBAC checks.

## 7. Auth & roles

| Role           | Can submit | Can verify | Can approve | Can manage users |
| -------------- | :--------: | :--------: | :---------: | :--------------: |
| `super_admin`  | ✓          | ✓          | ✓           | ✓ (incl. admins) |
| `admin`        | ✓          | ✓          | ✓           | ✓ (non-admins)   |
| `moderator`    | ✓          | ✓          | —           | —                |
| `contributor`  | ✓          | —          | —           | —                |
| `viewer`       | —          | —          | —           | —                |
| anonymous      | —          | —          | —           | —                |

Public reads (cases, persons, public attachments) are unauthenticated.

Auth flow:

1. Email + password registration → user starts in `pending` status.
2. Admin approves via the admin panel → status becomes `approved`.
3. Login returns a short-lived **access JWT** (15 min) + a **refresh token**
   (30 days) stored as a hashed row in `sessions` and in an httpOnly cookie.

## 8. Frontend layout (`frontend/`)

- **React 18 + Vite + TypeScript + Tailwind**.
- **Routing** via `react-router-dom` v6.
- **i18n** via `i18next` + `react-i18next`. Bengali is the default and the
  fallback; English is the secondary language. The `<html lang>` attribute is
  updated whenever the user switches.
- **Fonts**: `Hind Siliguri` for Bengali UI, `Inter` for Latin UI, `Tiro
  Bangla` for display headlines.
- **API client** is a thin `fetch` wrapper in `src/lib/api.ts` reading
  `VITE_API_BASE_URL`.
- **State**: local component state for now; Zustand or TanStack Query will
  be added once we wire real endpoints. Auth state will live in a small
  context.

## 9. Local development

```bash
cp .env.example .env
docker compose up --build
```

- Postgres 18 → `localhost:5432`
- MinIO (R2 substitute) → `localhost:9000` (API), `:9001` (console)
- Backend → `localhost:8080` (`/health` returns DB status)
- Frontend → `localhost:5173`

For migrations during early development (until the migrator is wired in):

```bash
docker compose exec -T postgres \
  psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" \
  < backend/migrations/0001_init.up.sql
```

## 10. Production deployment (planned)

- Single VPS or small fleet behind Cloudflare; Docker images for backend and
  frontend pushed to a registry.
- Postgres: managed (Neon/Supabase Postgres) or self-hosted with PITR backups.
- R2 for object storage; backups via R2 lifecycle + scheduled `aws s3 sync`
  to a cold archive.
- Observability: structured JSON logs (slog) → log aggregator; basic Prometheus
  metrics later.

## 11. Security & legal posture

- Rate-limit submissions and login attempts at the edge and at the API.
- Verification team must attach evidence before a case can be published.
- Submitter identity is **never public** — only admins can see who submitted
  what; surfaced in the admin panel and in `audit_logs`.
- Hidden / internal attachments are never served from the public CDN; access
  requires an authenticated admin and a fresh presigned URL.
- We will publish a takedown / dispute process before the first public
  release.

## 12. Open questions

- **Comments / corrections** on cases — open them up, or keep the archive
  read-only for the public?
- **NID handling** — we store only a hash, but should we also drop the
  hash entirely and keep just last 4 digits?
- **Cross-references** between cases (e.g. same accused, same location) —
  build a graph view in v2?
- **Backups & disaster recovery** — concrete RPO/RTO targets are TBD.
